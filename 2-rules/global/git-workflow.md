# Git Workflow Rules (3-Branch Strategy)

**Core Goal:** Maintain code integrity and environment separation through a strict branching and commit strategy.

## 1. Branch Structure
1. **`develop` (Development):** The primary workspace. 
   - All new features and bug fixes are implemented here. 
   - Uses local environment variables (`.env.local`).
2. **`main` (Integration):** The source of truth for stable code.
   - Only merges from `develop` via Pull Request/Merge Request.
   - Represents the "Ready for Review" state.
3. **`deploy` (Production):** The deployment-ready branch.
   - Contains production-specific environment variables and configurations.
   - Only merges from `main` after successful integration testing.

## 2. Commit Standards (Conventional Commits)
All commits generated by an agent must follow this format:
`type(scope): brief description`

- **feat:** A new feature for the user.
- **fix:** A bug fix.
- **docs:** Documentation only changes.
- **style:** Changes that do not affect the meaning of the code (white-space, formatting).
- **refactor:** A code change that neither fixes a bug nor adds a feature.

## 3. Feature Branch Naming

Work never happens directly on `develop`. Create a short-lived branch for each spec, then merge via PR.

| Type | Pattern | Example |
|------|---------|---------|
| New feature | `feat/<scope>-<description>` | `feat/users-registration` |
| Bug fix | `fix/<scope>-<description>` | `fix/checkout-total-calculation` |
| Refactor | `refactor/<scope>-<description>` | `refactor/orders-service-layer` |
| Docs | `docs/<description>` | `docs/update-api-readme` |

Rules:
- Use **kebab-case** only. No slashes beyond the type prefix.
- Keep descriptions short (2–4 words max).
- Branch off `develop`. Merge back into `develop` via PR.

## 4. Environment Variable Safety (CRITICAL)
- **NEVER** commit `.env` files to Git. 
- The agent must only modify `.env.example` or documentation when new variables are required.
- **`develop` branch:** Always assume `DEBUG=True` and local DB.
- **`deploy` branch:** Always assume `DEBUG=False` and production DB.
